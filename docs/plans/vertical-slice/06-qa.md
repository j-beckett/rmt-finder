# Manual QA — full-system verification

**Type:** HITL

## Parent PRD

`docs/prd/vertical-slice.md`

## What to build

Nothing new — this is the final human verification pass over the assembled
slice: scheduler + API + frontend dev server running together on the local
machine, walked through every state and checked against the PRD's
non-negotiable portability criteria.

## Manual QA plan

### Setup

1. Start from a clean checkout with no pre-existing DB file.
2. Start the scheduler, the API (uvicorn), and the frontend dev server as
   three processes.

### Pipeline & scheduling

3. Confirm the scheduler scraped immediately on startup: a run row and
   slots exist in SQLite, and the CLI-style summary appeared in its logs.
4. Wait one interval (temporarily set `SCRAPE_INTERVAL_MINUTES=1` to keep
   this fast) and confirm a second run row appears; confirm earlier rows
   are untouched (append-only).
5. Run the one-shot scrape CLI by hand while the scheduler is running;
   confirm it works standalone and prints run id, clinics scraped,
   failures, slot count.
6. Check the run rows record attempted/succeeded counts and failed clinic
   names matching the logs.

### API

7. `curl` `GET /api/availability`: slots match the latest run in SQLite;
   envelope carries served `scraped_at`, latest attempt, clinics
   attempted, failed clinics.
8. Confirm the city query param is accepted.

### Frontend — happy path

9. Page lists real slots soonest-first; spot-check three slots against the
   actual clinic booking pages (time shown matches clinic-local time; Book
   link lands on the right clinic).
10. "Last updated X ago" is accurate and ticks over after the next
    scheduled run.

### Frontend — states (drive with crafted DBs or stubbed API responses)

11. Staleness: stop the scheduler, wait past ~2× interval (or backdate the
    run row in a scratch DB) — loud banner appears.
12. Partial data: a run with some failed clinics — subtle "N of M clinics
    couldn't be checked" notice, visibly quieter than the staleness banner.
13. Total failure: latest attempt has zero successes — page shows the last
    good run's data plus the "last check failed — showing results from
    HH:MM" notice.
14. Genuine empty: a successful run with zero slots — friendly empty state,
    clearly not an error.

### Resilience

15. Kill network access (or point one clinic at a bad subdomain in a
    scratch config) mid-schedule: the failing run is logged and recorded,
    and the next cycle still fires.

### Portability acceptance criteria (verbatim from the PRD — check each)

16. All config/paths via env vars with sensible local defaults — move the
    DB path and change the interval via env alone; confirm no code edits
    needed.
17. One storage module owns all DB access (SQLite now, swappable later) —
    grep-level review: no SQL/connection outside the storage module.
18. Scrape runs as a one-shot CLI command, schedulable by anything —
    verified in step 5.
19. Frontend API base URL is configurable, not hardcoded — change it via
    env, confirm the frontend follows.

### Suites

20. `pytest` passes; Vitest passes.
21. Confirm no JSON files are written to `data/` by any path.
